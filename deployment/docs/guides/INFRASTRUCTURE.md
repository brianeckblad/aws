# Infrastructure Reference

AWS resources created by this repo — what they are, why they exist, and how they are named.

---

## Resource Map

| AWS Resource | Name | Purpose |
|-------------|------|---------|
| EC2 instance | `{host_name}` (Name tag) | The shared application server |
| Security group | `{host_name}-sg` | Virtual firewall — ports 22, 80, 443 |
| IAM role | `{host_name}-ec2-role` | Grants EC2 access to AWS services |
| Instance profile | `{host_name}-instance-profile` | Attaches IAM role to EC2 |
| SSH key pair | `{host_name}-key` | Saved to `~/.ssh/{host_name}-key.pem` |
| EBS root volume | (auto-named) | OS disk — deleted on termination |
| EBS data volume | `{host_name}-data` | App data — **survives termination** |

`host_name` is set in `group_vars/vault.yml`. All AWS resource names derive from it — change it there and the names change everywhere.

## EC2 Instance

**OS:** Ubuntu 24.04 LTS  
**Default type:** `t3.micro` (configurable via `ec2_instance_type`)  
**AMI:** Latest Ubuntu 24.04 LTS in the specified region (resolved at deploy time)

### Why t3.micro?

Nitro-based instance types (t3, t3a, m5, c5, r5, and later families) expose EBS volumes as NVMe devices (`/dev/nvme1n1`). The EBS mount logic in `harden-server.yml` expects NVMe paths. If you switch to an older type (t2, m4), update `ebs_nvme_device` in vault.yml to match the actual device path.

`t3.micro` works for light loads and development. Upgrade to `t3.small` or larger for 2–4 production apps. Resize by changing `ec2_instance_type` in vault and launching a new instance (or resizing via the console — no playbook change needed).

### Tags

```
Name:      {host_name}
Server:    {host_name}
AppsRoot:  {apps_root}   (e.g. /opt/apps)
ManagedBy: ansible
```

### Termination protection

`ec2_termination_protection: true` — prevents accidental `aws ec2 terminate-instances` calls from the CLI. The `terminate-ec2-instance.yml` playbook disables this before terminating.

## Security Group

**Name:** `{host_name}-sg`

**Ingress rules:**

| Port | Protocol | Source | Purpose |
|------|----------|--------|---------|
| 22 | TCP | `0.0.0.0/0` | SSH for deployment |
| 80 | TCP | `0.0.0.0/0` | HTTP (nginx redirects to HTTPS) |
| 443 | TCP | `0.0.0.0/0` | HTTPS |

**Egress:** All traffic allowed (required for apt updates and Let's Encrypt)

> SSH is open to the internet because deployments run from anywhere. The effective SSH lockdown is on the host — public key only, no passwords, fail2ban bans after 3 failed attempts.

**Defense-in-depth:** UFW on the host enforces the same three ports at the OS level, independent of the AWS SG. An attacker who somehow bypassed the SG would still be blocked by UFW.

## IAM Role

**Name:** `{host_name}-ec2-role`  
**Type:** EC2 instance role (no user credentials stored on the server)

### Inline policies

| Policy name | What it allows |
|-------------|---------------|
| `{host_name}-s3-access` | Read/write to all S3 buckets (`arn:aws:s3:::*/*`) — covers every app on the server without per-app IAM changes |
| `{host_name}-secrets-access` | Read all Secrets Manager secrets in the region (`arn:aws:secretsmanager:{region}:*:secret:*`) |
| `{host_name}-cloudwatch-access` | Put metrics, log events, describe alarms |
| `{host_name}-sns-access` | Publish to SNS topics |
| `{host_name}-serial-console-access` | EC2 serial console (emergency access if SSH is unreachable) |

### Managed policies

| Policy | Why |
|--------|-----|
| `AmazonSSMManagedInstanceCore` | AWS Systems Manager — session manager, patch manager, run command |

### Why broad S3 and Secrets Manager scope?

This server hosts multiple applications. Each app owns its own bucket and secret path (e.g., `myapp/production`). Locking the role to a single app's ARNs would require a role update every time a new app is added. The wildcard policies let any app on the server access its own resources using its app-specific paths without infrastructure changes.

Application-level isolation is enforced by the name convention and IAM conditions in each app's code, not by restricting the instance role.

## SSH Key Pair

**Name:** `{host_name}-key`  
**Local path:** `~/.ssh/{host_name}-key.pem` (permissions set to `0600` by the playbook)

The key is generated by AWS and stored locally. It is never uploaded to any server — EC2 automatically injects the public key into `~ubuntu/.ssh/authorized_keys` at first boot.

To add additional operators, add their public keys to `~ubuntu/.ssh/authorized_keys` on the server — no playbook change needed.

## EBS Volumes

### Root volume (boot disk)

| Setting | Value |
|---------|-------|
| Type | gp3 |
| Size | 20 GB (Ubuntu default) |
| Deleted on termination | Yes |

The root volume holds the OS, packages, nginx config, supervisor config, and SSL certificates. These are all managed by Ansible and can be recreated from playbooks — no persistent data lives here.

### Data volume (app data)

| Setting | Value |
|---------|-------|
| Name tag | `{host_name}-data` |
| Type | gp3 |
| Size | `ebs_volume_size` (default: 100 GB) |
| Encrypted | Yes (`ebs_volume_encrypted: true`) |
| Deleted on termination | **No** — survives EC2 termination |
| Filesystem | XFS (formatted on first run only) |
| Mount point | `apps_root` (`/opt/apps`) |

The data volume is formatted and mounted by `harden-server.yml`. On subsequent runs, `blkid` detects the existing filesystem and skips formatting — data is never lost on re-runs. The UUID is persisted to `/etc/fstab` so the mount survives reboots.

**Snapshot restore:** Set `ebs_volume_snapshot_id` in vault.yml to restore an existing data volume when launching a replacement instance. The snapshot contains all app directories from `/opt/apps`.

### App directory layout on the data volume

```
/opt/apps/                 ← EBS data volume mount (apps_root)
├── appname1/              ← deployed by appname1's setup.yml
│   ├── .venv/
│   ├── app/
│   └── instance/
├── appname2/
│   └── ...
└── ...

/var/log/apps/             ← log root (logs_root) — on root volume
├── appname1/
│   └── app.log
└── appname2/
    └── app.log
```
